# Техническое задание
## Система NPC, поведения и боевой архитектуры

**Документ для Claude Code. Версия 1.0**  
Контекст: изометрическая tile-based игра с большим количеством персонажей, lane-based движением, точками интереса и перспективой добавления боевой системы.

> Ключевые корректировки: архитектура должна изначально поддерживать бой, а также систему постоянных черт и временных модификаторов состояния.

Цель документа — дать Claude Code не просто задачу на «ходячих NPC», а полноценный каркас симуляции, который хорошо масштабируется на десятки и сотни персонажей, остаётся дешёвым по CPU/GC и при этом не ломается, когда в проект позже добавляются бой, агро, статусы, черты характера и различия между типами персонажей.

## 1. Контекст и целевое направление

Игра строится вокруг изометрической локации на тайлах, где по заранее определённым дорожкам и зонам двигается много NPC. Базовое ощущение должно быть близко к cozy / management играм: персонажи создают живой поток, заходят к точкам интереса, ждут, взаимодействуют, иногда меняют маршрут и покидают сцену. При этом архитектура не должна быть «одноразовой»: в неё сразу нужно заложить боевой слой и влияние характеров/состояний на поведение.

Главная инженерная задача — не построить максимально умный AI, а сделать систему, которая выглядит правдоподобно, понятно расширяется и выдерживает рост сложности без полного рефакторинга через несколько месяцев.

## 2. Главные требования к архитектуре

- Движение должно опираться на lane graph / waypoints / tile paths, а не на тяжёлый realtime pathfinding каждый кадр.
- Исполнение поведения нужно строить через иерархическую или двухслойную state machine: отдельно выбор намерения, отдельно исполнение состояния.
- Система должна с первого дня быть combat-ready: гражданское поведение, реакция на угрозу и боевые действия не должны смешиваться в один неуправляемый enum.
- Характеристики, черты и модификаторы должны быть data-driven и влиять на выбор решений, скорость, терпение, агрессию, приоритет целей и исход боевых/небоевых взаимодействий.
- Нужен центральный manager, который управляет спавном, пулом объектов, тиками логики, LOD симуляции и бюджетом обновлений.
- Нужно избегать архитектуры, которая требует полного пересчёта маршрута, восприятия и принятия решения у каждого NPC каждый кадр.

## 3. Рекомендуемая модель AI: не только FSM, а Decision Layer + Execution Layer

Для текущей задачи простой плоской FSM уже недостаточно, потому что в проекте появятся бой, статусы и разные архетипы персонажей. Поэтому рекомендуется двухслойная модель:

- **Decision Layer** — дешёвый слой принятия решения. Он периодически оценивает доступные цели и выбирает намерение: идти дальше, посетить точку интереса, стоять в очереди, атаковать, убегать, вернуться к маршруту и т.д.
- **Execution Layer** — state machine, которая исполняет выбранное намерение: `Walk`, `MoveToTarget`, `Queue`, `Interact`, `Chase`, `Attack`, `Flee`, `Recover` и другие состояния.
- **Combat Interrupt Layer** — набор глобальных прерываний: `Stunned`, `KnockedDown`, `Dead`, `ForcedRetreat`, `Fear` и т.п. Этот слой временно перехватывает управление у обычного поведения.

Такой подход остаётся дешёвым, как и обычная FSM, но лучше переносит рост проекта: civilian logic, combat logic и системные прерывания перестают конфликтовать друг с другом.

## 4. Высокоуровневая архитектура модулей

| Модуль | Назначение | Ответственность |
|---|---|---|
| `NPCEntity` | Runtime-объект NPC | Хранит ссылки на компоненты, runtime id, фракцию, архетип, жизненный цикл и общее состояние. |
| `NPCBrain` | Принятие решений | Считает utility / weights, выбирает намерение, реагирует на угрозы, очереди, занятость точек интереса и персональные модификаторы. |
| `NPCStateMachine` | Исполнение поведения | Переключает конкретные состояния исполнения и обрабатывает переходы, `enter/update/exit`, прерывания и возврат к рутине. |
| `NPCMovement` | Передвижение | Движение по lane graph / waypoints, follow target, stop distance, локальное avoidance и возврат в маршрут. |
| `NPCCombatController` | Боевой слой | `Target selection`, attack cycle, range checks, cooldowns, hit reactions, retreat/flee, hooks для урона и статусов. |
| `NPCStatsComponent` | Характеристики | Агрегирует базовые характеристики, постоянные черты, временные модификаторы и вычисляет итоговые значения. |
| `NPCPerception` | Восприятие | Дешёвая проверка близких объектов, врагов, POI, очередей и триггеров опасности по тикам, а не каждый кадр. |
| `NPCManager / CrowdManager` | Оркестратор толпы | `Spawn/despawn`, pooling, тикеты обновления, лимиты NPC, LOD симуляции, профили обновления. |
| `PointOfInterest` | Игровая точка интереса | `Capacity`, queue slots, `reserve/release`, interaction rules, возможные боевые/небоевые эффекты. |

## 5. Data-driven модель персонажа

Чтобы персонажи с разными типами поведения не превращались в набор `if/else` по всему проекту, данные нужно разделить на три слоя: архетип, постоянные черты и временные модификаторы.

### 5.1. Архетип (`Archetype / NPCProfile`)

- Базовый набор характеристик: скорость, здоровье, выносливость, радиус восприятия, терпение, склонность к риску, агрессия, жадность, дисциплина, шанс зайти к POI.
- Предпочтительные активности: что персонаж любит делать в мирном режиме и какие действия чаще выбирает в боевом режиме.
- Разрешённые наборы состояний: например, один тип может уметь `Flee`, другой — `Guard`, третий — `Support`.
- Набор тегов и фракция: `civilian`, `merchant`, `hostile`, `guard`, `neutral` и т.д.

### 5.2. Постоянные черты (`Traits`)

Черты — это относительно постоянные особенности персонажа, которые редко меняются и должны переживать сессии/сцены.

- Примеры: жадный, осторожный, трусливый, вспыльчивый, дисциплинированный, добрый, мстительный.
- Черты не обязаны напрямую хранить цифры; лучше, чтобы они давали набор модификаторов или влияний на utility scoring.
- Пример: черта «жадный» повышает вес выбора выгодных POI, уменьшает готовность делиться ресурсом и может повышать шанс остаться в очереди дольше обычного.

### 5.3. Временные модификаторы (`Modifiers / Statuses`)

Модификаторы — это динамические состояния, которые появляются и исчезают во время игры.

- Примеры: устал, расстроен, вдохновлён, ранен, напуган, оглушён, разозлён, голоден.
- Каждый модификатор должен иметь источник, длительность, стекуемость, набор тегов, приоритет и набор влияний на `stats / decisions / combat`.
- Модификатор может влиять не только на цифры, но и на разрешённые переходы между состояниями. Например, «оглушён» временно блокирует `Walk/Attack` и переводит в глобальное interrupt-состояние.

### 5.4. Как черты и модификаторы влияют на поведение

| Тип данных | Пример | На что влияет | Пример эффекта |
|---|---|---|---|
| `Trait` | Жадный | `Utility scoring`, очередь, выбор POI | Выше шанс пойти к торговой точке, дольше терпит очередь, хуже реагирует на просьбы союзников. |
| `Trait` | Трусливый | Бой, реакция на угрозу | Раньше выбирает `Flee`, позже входит в `Chase/Engage`, быстрее теряет morale. |
| `Modifier` | Устал | Скорость, выносливость, решения | Ниже скорость, чаще выбирает отдых, реже начинает бой первым. |
| `Modifier` | Расстроен | Социальные и боевые реакции | Может игнорировать POI, сильнее реагировать на провокации, хуже ждать в очереди. |
| `Modifier` | Оглушён | `Execution layer` | Перехватывает state machine, временно запрещает действия и движение. |

Рекомендуемый принцип: **traits** задают долгосрочный «характер» NPC, а **modifiers** меняют текущее поведение поверх этого характера. Итоговое решение NPC должно рождаться не из жёстких `if/else`, а из комбинации базового профиля, черт, текущих статусов и контекста сцены.

## 6. Структура state machine с учётом будущего боя

Не рекомендуется делать одну плоскую FSM вида `Spawn/Walk/Wait/Attack/Death/Queue/...` — она быстро станет хрупкой. Ниже рекомендуемая разбивка по доменам.

### 6.1. Базовые мирные состояния

- `Spawn` — инициализация, получение профиля, назначение маршрута и начальных модификаторов.
- `EnterLane` — вход в поток движения / присоединение к маршруту.
- `Walk` — основное движение по lane path.
- `ChooseTarget` — выбор ближайшей или наиболее выгодной цели.
- `MoveToTarget` — движение к POI, точке сбора, месту отдыха или другой задаче.
- `Queue` — ожидание у занятой цели.
- `Interact` — мирное взаимодействие: покупка, осмотр, отдых, разговор, работа.
- `Wait / Idle` — краткая пауза или бездействие.
- `Leave` — переход к выходному маршруту.
- `Despawn` — отключение и возврат в пул.

### 6.2. Боевые состояния

- `Alert` — обнаружена угроза, но NPC ещё не вступил в бой.
- `Engage` — вход в боевой режим и подтверждение цели.
- `Chase / CloseDistance` — сближение с целью.
- `Attack` — исполнение атаки / серии атак / способности.
- `Recover` — короткое восстановление после действия или при нехватке ресурса.
- `Guard / HoldPosition` — удержание точки или ожидание в боевой готовности.
- `Flee / Retreat` — попытка выйти из боя при определённых условиях.
- `ReturnToRoutine` — выход из боевого режима и возвращение к мирному намерению.

### 6.3. Глобальные прерывания

- `Stunned / Disabled`
- `Knockback / ForcedMove`
- `Dead`
- `ScriptedControl / CutsceneLock`

Глобальные прерывания должны иметь более высокий приоритет, чем обычные состояния. После их завершения NPC либо возвращается в предыдущее намерение, либо делает новый выбор через `Decision Layer`.

## 7. Боевая система: что нужно заложить заранее

- Разделить понятия «намерение» и «анимация/действие». Персонаж может хотеть атаковать, но конкретное боевое действие исполняется отдельно и может быть отменено или прервано.
- Вынести боевые параметры в отдельные конфиги: `attack range`, `cooldown`, `windup`, `recovery`, `damage profile`, `allowed targets`, `aggro rules`.
- Поддержать `threat/aggro model` даже в упрощённом виде: кто вызвал реакцию, почему NPC сменил цель, когда он сбрасывает агро и возвращается к мирному поведению.
- Предусмотреть события для боевой системы: `OnDamageTaken`, `OnTargetLost`, `OnStateInterrupted`, `OnCombatStarted`, `OnCombatEnded`.
- Не смешивать навигацию и бой в одном компоненте: `NPCMovement` должен уметь двигать персонажа, а `NPCCombatController` — только запрашивать нужный тип движения.

## 8. Навигация и движение толпы

Для данной игры рекомендуется lane-based навигация. Уровень должен задавать несколько предсказуемых путей и узлов, между которыми NPC могут переключаться по правилам, не запуская дорогой поиск пути на каждый шаг.

- Использовать `waypoints / lane graph / tile path graph`.
- Пересчитывать маршрут только при смене цели, блокировке или боевом манёвре, а не на каждом кадре.
- `Avoidance` сделать дешёвым: дистанция до впереди идущего, снижение скорости, краткий stop, возможная смена lane/slot там, где это разрешено.
- В бою `movement` должен поддерживать разные режимы: `route following`, `pursuit`, `strafe/hold position`, `retreat`.

## 9. Производительность и масштабирование

- `Object pooling` обязателен для NPC, VFX-объектов, простых индикаторов и временных боевых сущностей.
- Логику нужно обновлять тиками и бакетами: не все NPC пересчитывают `decision/perception` в один и тот же кадр.
- `Visual movement` может идти каждый кадр, но `perception/decision/combat think` — реже, например 4–10 раз в секунду по профилям.
- Нужны уровни детализации симуляции: off-screen NPC могут работать в упрощённом режиме без полного набора проверок.
- Избегать `LINQ`, частых allocations, бесконтрольных `GetComponent` и событий с лишним `boxing/unboxing` в горячих местах.

## 10. Минимальный MVP и последовательность внедрения

| Этап | Что реализовать | Зачем это нужно |
|---|---|---|
| `MVP-1` | Pooling, `spawn/despawn`, lane movement, `Walk/Wait/Leave/Despawn` | Проверить базовый поток NPC и нагрузку системы. |
| `MVP-2` | `Decision Layer`, `POI`, `Queue`, `Interact`, простые черты и модификаторы | Закрепить data-driven поведение до добавления боя. |
| `MVP-3` | `Perception`, `Alert`, `Engage`, `Chase`, `Attack`, `ReturnToRoutine` | Ввести минимальный боевой цикл без полной тактической сложности. |
| `MVP-4` | `Status effects`, interrupt states, combat events, threat reset | Сделать архитектуру устойчивой к росту боевой логики. |
| `MVP-5` | `LOD simulation`, debug tools, profiling, tuning | Подготовить систему к реальному наполнению контентом. |


## 12. Готовый текст задачи для Claude Code

```text
Нужно спроектировать и подготовить production-friendly архитектуру NPC system для изометрической tile-based игры с большим количеством персонажей. Игра по ощущениям близка к cozy / management flow, но архитектура должна с первого дня поддерживать боевую систему.

Контекст:
- карта из тайлов, движение в основном по lane paths / waypoint graph;
- на сцене одновременно может быть много NPC;
- персонажи ходят, выбирают точки интереса, ждут, взаимодействуют, покидают сцену;
- позже и/или параллельно у них будет бой, реакции на угрозу, статусы и возврат к рутине.

Ключевые требования:
1. Не строить систему вокруг тяжёлого realtime pathfinding каждый кадр.
2. Использовать lane-based movement, predefined paths и дешёвое avoidance.
3. Использовать не плоскую FSM, а layered approach:
   - Decision Layer для выбора намерения;
   - Execution State Machine для исполнения;
   - global interrupts для статусов и смерти.
4. Сразу заложить поддержку боевых состояний и переходов: Alert, Engage, Chase, Attack, Recover, Flee, ReturnToRoutine.
5. Разделить character data на:
   - archetype / profile;
   - permanent traits (например: жадный, трусливый);
   - temporary modifiers/statuses (например: устал, расстроен, оглушён, ранен).
6. Traits и modifiers должны влиять на utility scoring, transitions, скорость, терпение, агрессию, выбор целей и боевое поведение.
7. Нужен NPCManager/CrowdManager с pooling, tick-based updates, LOD simulation и лимитом активных NPC.
8. Нужны PointOfInterest с queue/capacity/reserve/release.
9. Нужна производительная архитектура для десятков/сотен NPC без лишнего GC.

Что нужно выдать:
- архитектуру системы;
- список классов и ответственности;
- схему данных;
- пример layered state machine;
- пример utility-based decision logic;
- пример кода на C# для Unity;
- поэтапный план внедрения от MVP до расширенной версии;
- рекомендации по производительности и масштабированию.

Важно:
- не переусложнять AI;
- делать систему расширяемой;
- сразу закладывать совместимость мирного поведения и боя;
- учитывать, что разные типы персонажей имеют разные traits/modifiers и это должно быть частью архитектуры, а не костылём поверх неё.
```
